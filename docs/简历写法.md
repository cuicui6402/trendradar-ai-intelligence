## TrendRadar 简历写法参考

> 本文档提供简历中 TrendRadar 项目的写法建议，可根据实际岗位和个人情况调整。

---

### 项目名称

TrendRadar — 面向求职与技术选型的 AI 趋势情报平台

### 一句话描述

基于 GitHub 和 Hacker News 多源数据采集，设计趋势评分模型和 AI 报告 Agent，生成面向求职选题和技术调研的结构化分析报告。

### 技术栈写法

Python、Streamlit、Pandas、SQLite/SQLAlchemy、Plotly、Requests、OpenAI 兼容 API

### 项目职责写法（参考）

以下按功能模块拆分，选取 3-4 条写进简历即可：

- 设计并实现 GitHub 和 Hacker News 双源数据采集模块，调用公开 API 采集近期活跃数据，实现 3 级 fallback 查询策略提高数据召回率
- 设计多维度加权趋势评分模型，综合 Stars、Forks、Issues、Points、评论数和时间新鲜度计算 0-100 趋势评分，区分"当前热门"和"历史热门"
- 实现 AI 趋势报告 Agent，支持无需 API Key 的模板模式和可选的 LLM 深度分析模式，生成包含 9 个章节的 Markdown 报告
- 设计 GitHub 项目相关性评分算法，通过关键词匹配和资料类项目识别，过滤面试指南、算法题库等无关项目
- 使用 SQLite + SQLAlchemy 实现搜索历史记录，每次分析自动持久化，支持历史查询和报告回溯
- 编写自动化测试，覆盖评分逻辑、数据库操作、Agent 报告生成和相关性筛选等核心模块

### 项目亮点写法（选 2-3 条）

**亮点 1：多源数据采集 + 多级 fallback**
两个数据源（GitHub + HN）独立采集，单个源失败不影响另一个。GitHub 采集器实现了 3 级 fallback（精确匹配 + 限定时间 → 名称/描述/标签匹配 → 简化关键词 + 不限时间），在 API 限流和网络不稳定时仍能保证数据可用性。

**亮点 2：趋势评分模型**
不是简单按 Stars 排序，而是综合多个维度加权评分。时间新鲜度模块根据距离当前日期的天数做分段线性衰减（今天 100 → 7 天 90 → 30 天 50 → 90 天 10），让近期活跃的项目排名更靠前。

**亮点 3：报告相关性筛选**
报告中展示的 GitHub Top 5 项目经过相关性评分，repo_name 和 description 命中关键词相关词加分，资料类项目（interview/guide/leetcode/awesome）降权或过滤，保证报告内容与搜索关键词高度相关。

**亮点 4：双模式 Agent 设计**
模板模式保证无 API Key 时也能生成完整报告，LLM 模式增强分析深度。LLM 调用失败时自动回退到模板模式，保证功能可用性。

### 面试官可能追问的问题

**Q: 为什么用 min-max 归一化而不是 z-score？**
A: 因为评分结果需要映射到 0-100 的固定区间，min-max 归一化能保证结果在确定的范围内。z-score 的结果没有固定上下界，不适合这种需要直观展示的场景。数据量不大（每次最多 20 条），min-max 对异常值的敏感度在这里不是问题。

**Q: 评分权重怎么确定的？有数据支撑吗？**
A: 第一版权重是基于经验设定的。Stars 占 40% 是因为它仍然是 GitHub 上最直观的热度指标，新鲜度占 30% 是为了区分当前活跃和历史热门。目前没有用真实数据做回归验证，这是后续可以优化的方向——可以用历史数据做 A/B 测试或回归分析来调参。

**Q: fallback 查询会影响结果准确性吗？**
A: 会有影响。broad 级别的结果可能包含与原始关键词不完全相关的项目。但我在前端会显示当前查询级别的提示，让用户知道结果范围已经扩大。报告中还有相关性评分，会优先展示相关度高的项目。

**Q: 数据源只有 GitHub 和 HN，会不会太少了？**
A: 确实是目前的一个局限。GitHub 和 HN 是英文社区的代表性数据源，对英文技术方向的覆盖面较好。国内数据源（V2EX、掘金等）是后续优化方向。目前的 2 个数据源已经能验证整套采集→评分→报告的流程。

**Q: Agent 的模板模式和 LLM 模式怎么切换的？**
A: 通过环境变量 LLM_API_KEY 判断。有这个 key 就用 LLM 模式，没有就用模板模式。LLM 模式内部有 try-except，如果 API 调用失败（超时、限流、网络错误），自动回退到模板模式。用户始终能看到报告，只是内容深度不同。

**Q: 如果 GitHub API 限流了怎么办？**
A: 未认证请求的频率受到 GitHub 官方限制。我的采集器会捕获 403/429 状态码，在前端显示"API 限流"的提示，并建议用户配置 GITHUB_TOKEN（个人访问令牌），配置后限制会大幅提升。同时 fallback 机制会减少不必要的重复请求。

**Q: SQLite 能支撑多大的数据量？**
A: 这是个人项目，数据量不大（每次搜索最多 20 条 GitHub + 20 条 HN），SQLite 完全够用。如果需要支撑更大规模的数据，可以迁移到 PostgreSQL，ORM 层的代码基本不需要改。
