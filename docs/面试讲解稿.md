## TrendRadar 面试讲解稿

> 以下用学生第一人称视角写，面试时可以按这个思路讲。每个模块不需要一次全讲，根据面试官的兴趣选择重点。

---

### 一、为什么做这个项目

准备求职的时候，我发现一个问题：不知道该选什么技术方向做项目。比如"AI Agent 现在火不火？""LangGraph 值不值得学？""FastAPI 是不是过时了？"这些问题光靠刷帖子很难判断，因为每个人说的都不一样。

我就想，能不能用数据来回答这个问题？GitHub 上有多少人在用、最近有没有更新、社区讨论热不热烈——这些都是可以量化的指标。

所以 TrendRadar 的初衷很简单：输入一个技术关键词，自动采集 GitHub 和 Hacker News 的数据，算一个趋势评分，生成一份报告，告诉我这个技术现在到底怎么样。

---

### 二、项目整体流程

用户在前端输入关键词（比如"AI Agent"），选择时间范围（比如近 30 天），点击分析。

系统会同时向 GitHub API 和 Hacker News API 发请求，拿到最近活跃的仓库和讨论数据。拿到数据后，用评分模型给每条数据算一个 0-100 的趋势评分。评分结果在前端用表格和柱状图展示。

用户还可以点击"生成报告"，系统会基于这些数据生成一份 9 章节的 Markdown 报告，包含技术概览、热度判断、项目解读、业务场景、求职推荐、面试问题和学习路径。报告自动保存到 data/reports 目录，同时在数据库里留一条记录。

每次搜索都会保存到 SQLite 数据库，侧边栏可以看到历史搜索记录。

---

### 三、核心模块怎么实现

**数据采集模块**在 collectors 目录下。GitHub 采集器调的是 GitHub Search Repositories API，HN 采集器调的是 Algolia 的 HN Search API。

两个采集器都继承了一个基类 BaseCollector，基类里封装了一个 `_safe_request` 方法，处理超时、HTTP 错误和重试逻辑。所有外部请求都经过这个方法，保证不会因为网络问题导致整个程序崩溃。

**多级 fallback 查询**是采集器的一个重点。以 GitHub 为例，用户搜"AI Agent 近 7 天"，系统先用精确查询（关键词 + 限定时间），如果没找到结果，自动放宽到"在仓库名称、描述和标签中匹配关键词"，再没有就简化为只搜关键词、不限时间。每一级查询的结果状态都会记录在 `last_result_meta` 字典里，前端可以根据这个信息给用户不同的提示。

**HN 采集器**也类似，两级 fallback：先搜近 N 天的 story，搜不到就搜全时段。

**前端**用 Streamlit，没有写复杂的业务逻辑，所有数据采集和评分都在 collectors 和 services 里完成。前端的 `_run_github_analysis` 和 `_run_hn_analysis` 两个函数负责调用采集器、展示结果、返回评分后的 DataFrame。

### 四、趋势评分模型怎么设计

最初我直接用 Stars 排序，但发现一个问题：一个 5 年前获得 10000 stars 的项目排在第一位，但它可能已经停止维护了。而一个上周刚发布但只有 200 stars 的项目可能正在快速增长。

所以我设计了一个多维度加权评分模型。

**GitHub 仓库**的趋势评分公式是：

```
trend_score = Stars × 0.4 + Forks × 0.2 + Issues × 0.1 + 新鲜度 × 0.3
```

每个维度先做归一化（0-100），用 min-max 归一化，把原始数值映射到 0-100 区间。

**时间新鲜度**是关键。规则是：今天 100 分，7 天内 90-100，30 天内 50-90，90 天内 10-50，超过 90 天只有 10 分。这样近期活跃的项目会比历史热门但已经沉寂的项目得到更高的分数。

**HN 帖子**的公式是：Points × 0.4 + 评论数 × 0.3 + 新鲜度 × 0.3。

权重是我根据经验设定的，Stars 和 Points 占最大权重是因为它们仍然是最直观的热度指标，但新鲜度占 30% 可以有效区分"当前热门"和"历史热门"。

### 五、Agent 报告怎么生成

报告 Agent 支持两种模式。

**模板模式**（默认，不需要 API Key）：根据采集到的数据，用规则模板生成 9 个章节的 Markdown 报告。包括技术概览、热度判断、GitHub 项目解读、HN 讨论分析、业务场景、国内生态、求职项目推荐、面试问题、学习路径。

这里有一个关键优化：**相关性筛选**。搜索"AI Agent"时，GitHub 返回的项目里可能混入 JavaGuide 这种面试指南（因为 Stars 特别多）。所以我在 Agent 里加了一个 `report_relevance_score`，对每个项目计算和关键词的相关性。repo_name 命中相关词 +3 分，description 命中 +2 分，资料类项目（包含 interview/guide/leetcode/awesome 等）-2 分。报告 Top 5 优先选相关性高的项目。

如果关键词包含"agent"，报告内容会围绕 AI Agent 展开：业务场景写智能客服、RAG 问答、数据分析 Agent 这些具体方向，求职项目推荐 DataInsight Agent、JobPilot 这类可落地的项目，面试问题覆盖 Tool Calling、任务规划、安全控制等考点。

**LLM 模式**（可选）：配置 LLM_API_KEY 后，Agent 会调用 OpenAI 兼容 API 生成深度报告。如果 LLM 调用失败，会自动回退到模板模式，保证用户始终能看到报告。

### 六、数据库历史记录怎么保存

用 SQLite + SQLAlchemy ORM。数据库有 4 张表：

- `search_history`：每次搜索记录关键词、时间范围、结果数量
- `github_results`：关联搜索记录的 GitHub 仓库评分结果
- `hn_results`：关联搜索记录的 HN 帖子评分结果
- `reports`：关联搜索记录的趋势报告

数据库在项目启动时自动初始化（`init_db`），不需要手动建表。保存操作在 `_save_to_database` 函数里完成，所有数据库操作都有 try-except，保存失败不会影响页面展示。

存取逻辑封装在 `repository.py` 里，app.py 只调用 repository 的函数，不直接写 ORM 操作。

### 七、遇到的问题和解决方法

**问题 1：GitHub API 经常搜不到结果**

用户搜"AI Agent 近 7 天"经常返回空，因为很多仓库的更新时间不一定在 7 天内。

解决：设计了 3 级 fallback 查询。strict（精确匹配 + 限定时间）→ fallback（在仓库名称、描述和标签中匹配）→ broad（简化关键词 + 不限时间）。每级查询的状态记录在 `last_result_meta` 里，前端根据这个信息显示不同的提示。

**问题 2：HN API 在国内偶尔超时**

解决：在基类 BaseCollector 里封装了 `_safe_request` 方法，统一处理超时（30 秒）、HTTP 错误、网络异常，失败时返回空 DataFrame 而不是抛异常。GitHub 失败不影响 HN 展示，两个源独立。

**问题 3：报告里混入无关项目**

搜索"AI Agent"时报告里出现 JavaGuide，因为它的 Stars 特别多，trend_score 排名靠前。

解决：在 TrendAgent 里加了相关性评分和资料类项目检测。用通用规则（interview/guide/leetcode/awesome 等关键词）识别资料类项目，在报告 Top 5 中优先选相关性高的项目。不是硬编码某个项目名，而是用通用的模式匹配。

**问题 4：数据库测试路径不一致**

`init_db(temp_path)` 创建表在一个引擎上，但 `get_session()` 用了默认路径。

解决：在 db.py 里用模块级变量 `_current_db_path` 统一记录当前路径，所有函数都引用这个变量。

### 八、项目不足和后续优化

说实话这个项目还有不少可以改进的地方：

1. **数据源单一**：目前只有 GitHub 和 HN，都是英文社区。国内数据源（V2EX、掘金、InfoQ）还没接入，对中文用户不够友好。
2. **评分权重固定**：权重是我根据经验设的，没有用真实数据验证。可以用历史数据做回归分析，找到更优的权重配比。
3. **没有缓存**：每次搜索都实时调 API，热门关键词会重复请求。可以加 Redis 缓存或本地缓存。
4. **报告质量依赖数据量**：如果搜索结果很少（比如小于 5 条），报告内容会显得空。可以通过扩大搜索范围或引入更多数据源来改善。
5. **部署方式简单**：目前是本地 `streamlit run`，没有做 Docker 部署和 CI/CD。

这些都是我在后续优化方向文档里记录的，如果有时间可以逐步实现。
